# Section 1 Hybrid Search

Hybrid Search is an advanced search technology that combines the advantages of Sparse Vectors and Dense Vectors. It aims to simultaneously utilize the precise keyword matching capabilities of sparse vectors and the semantic understanding capabilities of dense vectors to overcome the limitations of single vector retrieval, thereby providing more accurate and robust retrieval results in various search scenarios.

In this section, we will first analyze the characteristics of these two core vectors, then discuss how they are integrated, and finally implement hybrid retrieval through milvus.

## 1. Sparse vector vs dense vector

In order to better understand hybrid retrieval, we first need to clarify the essential difference between the two vectors.

### 1.1 Sparse vector

Sparse vectors, also often called "lexical vectors," are mathematical representations of traditional information retrieval methods based on word frequency statistics. It is usually a very high-dimensional vector (comparable to the size of the vocabulary) but with the vast majority of its elements being zero. It uses an exact "bag of words" matching model, treating a document as a collection of words without considering order or grammar. Each vector dimension directly corresponds to a specific word, and a nonzero value represents that word's importance or weight in the document. The classic weight calculation method for this type of vector is TF-IDF. In the field of information retrieval, BM25 is one of the successful and widely used sorting algorithms based on this sparse representation. Its core formula is as follows:

  $$ Score(Q, D) = \sum_{i=1}^{n} IDF(q_i) \cdot \frac{f(q_i, D) \cdot (k_1 + 1)}{f(q_i, D) + k_1 \cdot (1 - b + b \cdot \frac{|D|}{avgdl})} $$

in:
- $IDF(q_i)$: The inverse document frequency of query term $q_i$, used to measure the prevalence of a term. The more common words are, the lower the IDF value is.
- $f(q_i, D)$: Query the word frequency of word $q_i$ in document $D$.
- $|D|$: length of document $D$.
- $avgdl$: average length of all documents in the collection.
- $k_1, b$: adjustable hyperparameters. $k_1$ is used to control word frequency saturation (a word appears 10 times and 100 times in the document, its importance growth is not linear), $b$ is used to control the degree of document length normalization.

The advantage of this method is that it is highly interpretable (each dimension represents an exact word), does not require training, can achieve precise matching of keywords, and has good retrieval results for professional terms and specific nouns. The main disadvantage is the inability to understand semantics. For example, it cannot recognize that "car" and "car" are synonyms, and there is a "lexical gap."

### 1.2 Dense vectors

Dense vectors, also often called "semantic vectors", are low-dimensional, dense floating-point representations of data (such as text, images) learned by deep learning models. These vectors aim to capture "semantics" or "concepts" by mapping raw data into a continuous, meaning-filled "semantic space". In an ideal semantic space, the distance and direction between vectors represent the relationship between the concepts they represent. A classic example is that the calculation result of `vector('еӣҪзҺӢ') - vector('з”·дәә') + vector('еҘідәә')` is very close to `vector('еҘізҺӢ')` in vector space, which shows that the model has learned the abstract concepts of "gender" and "royalty". Its representatives include Word2Vec, GloVe, and all Embeddings generated by Transformer-based models (such as BERT, GPT).

Its main advantages are its ability to understand synonyms, synonyms and contextual relationships, its strong generalization ability, and its excellent performance in semantic search tasks. But the shortcomings are equally obvious: poor interpretability (each dimension in the vector usually has no specific physical meaning), a large amount of data and computing power are required for model training, and the processing of out-of-service words (OOV) [^1] is relatively difficult.

> **OOV (Out-of-Vocabulary) unregistered words**: refers to new words that do not appear in the vocabulary during model training but are encountered in actual use. For example, if the word "ChatGPT" is not in the vocabulary when the model is trained, it will be OOV when it is encountered in actual applications. Traditional sparse vector methods (such as BM25) will completely ignore OOV words, while modern dense vector methods can better handle OOV problems through sub-word segmentation (such as BPE, WordPiece).

### 1.3 Example comparison

**Sparse vector representation:**

The core idea of вҖӢвҖӢsparse vectors is to store only non-zero values. For example, an 8-dimensional vector `[0, 0, 0, 5, 0, 0, 0, 9]` has most of its elements zero. Represented in sparse format, space can be greatly saved. There are two common sparse representations:

1. **Dictionary / Key-Value:**
This approach uses nonzero elements `зҙўеј•` (0-based) as keys and `еҖј` as values. The above vector can be expressed as:
    ```json
    // {зҙўеј•: еҖј}
    {
      "3": 5,
      "7": 9
    }
    ```

2. **Coordinate list (Coordinate list - COO):**
This method is usually represented by a tuple `(з»ҙеәҰ, [зҙўеј•еҲ—иЎЁ], [еҖјеҲ—иЎЁ])`. The above vector can be expressed as:
    ```
    (8, [3, 7], [5, 9])
    ```
This format is very common in scientific computing libraries such as `SciPy`.

Assume that in a vocabulary containing 50,000 words, "tomato" is at position 88, "fried" is at position 666, and "egg" is at position 999. Their BM25 weights are 1.2, 0.8, and 1.5 respectively. Then its sparse representation (in dictionary format) is:
```json
// {зҙўеј•: жқғйҮҚ}
{
  "88": 1.2,
  "666": 0.8,
  "999": 1.5
}
```
If in coordinate list (COO) format it would look like this:
```
(50000, [88, 666, 999], [1.2, 0.8, 1.5])
```
Both formats clearly record the key information of the document, but their limitations are also obvious: if we search for "tomato scrambled eggs", since "tomato" and "tomato" are different terms (different indexes), the model will not be able to understand their semantic similarity.

**Dense vector representation:**

Unlike sparse vectors, dense vectors have values вҖӢвҖӢin all dimensions, so using an array `[]` to represent them is the most straightforward way. After reading "tomato scrambled eggs", a pre-trained semantic model will output a low-dimensional dense vector:

```json
// иҝҷжҳҜдёҖдёӘдҪҺз»ҙпјҲжҜ”еҰӮ1024з»ҙпјүзҡ„жө®зӮ№ж•°еҗ‘йҮҸ
// еҗ‘йҮҸзҡ„жҜҸдёӘз»ҙеәҰжІЎжңүзӣҙжҺҘзҡ„гҖҒеҸҜи§ЈйҮҠзҡ„еҗ«д№ү
[0.89, -0.12, 0.77, ..., -0.45]
```

This vector itself is difficult to interpret, but its position in the semantic space may be very close to the vectors of dishes such as "tomato egg noodles" and "fried eggs with onions" because the model understands that they share core concepts such as "egg dishes", "home-cooked dishes", and "sweet and sour taste". Therefore, when we search for "protein-rich home-cooked dishes", even if no original keywords appear in the query, the dense vector is very likely to successfully match this recipe.

## 2. Mixed search

From the above, we can see that sparse vectors and dense vectors have their own advantages, so it is a good choice to combine them to achieve complementary advantages. Hybrid retrieval is based on this idea, improving the relevance and recall of search results by combining multiple search algorithms (the most common ones are sparse and dense retrieval).

- **Main Objective**: Address the limitations of a single search technology. For example, keyword retrieval cannot understand semantics, while vector retrieval may ignore keywords that must be accurately matched (such as product model, function name, etc.). Hybrid retrieval aims to take advantage of both the **accuracy** of sparse vectors and the **generalization** of dense vectors to cope with complex and changing search needs.

### 2.1 Technical principles and integration methods

Hybrid retrieval typically executes two retrieval algorithms in parallel and then fuses the two sets of heterogeneous result sets into a unified sorted list. The following are two mainstream integration strategies:

#### 2.1.1 Reciprocal Rank Fusion (RRF)

RRF does not care about the raw scores of different retrieval systems, only the ranking of each document in its respective result set. The idea is: the higher a document ranks in different retrieval systems, the higher its final score.

The scoring formula is:

$$ RRF_{score}(d) = \sum_{i=1}^{k} \frac{1}{rank_i(d) + c} $$

in:
- $d$ is the document to be scored.
- $k$ is the number of retrieval systems (here 2, i.e. sparse and dense).
- $rank_i(d)$ is the ranking of document $d$ in the $i$th retrieval system.
- $c$ is a constant (usually set to 60) that is used to reduce the relative weight of top-ranked documents to achieve a more robust ranking fusion.

#### 2.1.2 Weighted linear combination

This method requires first normalizing the scores of different retrieval systems (for example, unifying them to the 0-1 interval), and then linearly combining them through a weight parameter `Оұ`.

$$ Hybrid_{score} = \alpha \cdot Dense_{score} + (1 - \alpha) \cdot Sparse_{score} $$

By adjusting the value of `Оұ`, you can flexibly control the proportion of semantic similarity and keyword matching in the final ranking. For example, in e-commerce search, you can increase the weight of keywords; in intelligent question and answer, you can focus on semantics.

### 2.2 Advantages and Limitations

| Advantages | Limitations |
| :--- | :--- |
| **High recall rate and accuracy**: It can capture keywords and semantics at the same time, which is significantly better than single retrieval. | **High computing resource consumption**: Two sets of indexes need to be maintained and queried at the same time. |
| **Strong flexibility**: It can be adapted to different business scenarios through integration strategies and weight adjustments. | **Complex parameter debugging**: Hyperparameters such as fusion weights require repeated experiments and tuning. |
| **Good fault tolerance**: Keyword retrieval can partially compensate for the sensitivity of vector models to spelling errors or rare words. | **Interpretability is still a challenge**: The reasons for ranking the results after fusion are difficult to analyze intuitively. |

## 3. Code practice: hybrid retrieval through Milvus

Next, use Milvus to implement a complete hybrid retrieval process, from defining Schema, inserting data, to executing queries.

### 3.1 Step 1: Define Collection

In the previous chapter, we implemented multi-modal image and text retrieval. Now we follow the same steps to create a Collection.

```python
import json
import os
os.environ["HF_ENDPOINT"] = "https://hf-mirror.com"
import numpy as np
from pymilvus import connections, MilvusClient, FieldSchema, CollectionSchema, DataType, Collection, AnnSearchRequest, RRFRanker
from pymilvus.model.hybrid import BGEM3EmbeddingFunction

# 1. еҲқе§ӢеҢ–и®ҫзҪ®
COLLECTION_NAME = "dragon_hybrid_demo"
MILVUS_URI = "http://localhost:19530"  # жңҚеҠЎеҷЁжЁЎејҸ
DATA_PATH = "../../data/C4/metadata/dragon.json"  # зӣёеҜ№и·Ҝеҫ„
BATCH_SIZE = 50

# 2. иҝһжҺҘ Milvus е№¶еҲқе§ӢеҢ–еөҢе…ҘжЁЎеһӢ
print(f"--> жӯЈеңЁиҝһжҺҘеҲ° Milvus: {MILVUS_URI}")
connections.connect(uri=MILVUS_URI)

print("--> жӯЈеңЁеҲқе§ӢеҢ– BGE-M3 еөҢе…ҘжЁЎеһӢ...")
ef = BGEM3EmbeddingFunction(use_fp16=False, device="cpu")
print(f"--> еөҢе…ҘжЁЎеһӢеҲқе§ӢеҢ–е®ҢжҲҗгҖӮеҜҶйӣҶеҗ‘йҮҸз»ҙеәҰ: {ef.dim['dense']}")

# 3. еҲӣе»ә Collection
milvus_client = MilvusClient(uri=MILVUS_URI)
if milvus_client.has_collection(COLLECTION_NAME):
    print(f"--> жӯЈеңЁеҲ йҷӨе·ІеӯҳеңЁзҡ„ Collection '{COLLECTION_NAME}'...")
    milvus_client.drop_collection(COLLECTION_NAME)

fields = [
    FieldSchema(name="pk", dtype=DataType.VARCHAR, is_primary=True, auto_id=True, max_length=100),
    FieldSchema(name="img_id", dtype=DataType.VARCHAR, max_length=100),
    FieldSchema(name="path", dtype=DataType.VARCHAR, max_length=256),
    FieldSchema(name="title", dtype=DataType.VARCHAR, max_length=256),
    FieldSchema(name="description", dtype=DataType.VARCHAR, max_length=4096),
    FieldSchema(name="category", dtype=DataType.VARCHAR, max_length=64),
    FieldSchema(name="location", dtype=DataType.VARCHAR, max_length=128),
    FieldSchema(name="environment", dtype=DataType.VARCHAR, max_length=64),
    FieldSchema(name="sparse_vector", dtype=DataType.SPARSE_FLOAT_VECTOR),
    FieldSchema(name="dense_vector", dtype=DataType.FLOAT_VECTOR, dim=ef.dim["dense"])
]

# еҰӮжһңйӣҶеҗҲдёҚеӯҳеңЁпјҢеҲҷеҲӣе»әе®ғеҸҠзҙўеј•
if not milvus_client.has_collection(COLLECTION_NAME):
    print(f"--> жӯЈеңЁеҲӣе»ә Collection '{COLLECTION_NAME}'...")
    schema = CollectionSchema(fields, description="е…ідәҺйҫҷзҡ„ж··еҗҲжЈҖзҙўзӨәдҫӢ")
    # еҲӣе»әйӣҶеҗҲ
    collection = Collection(name=COLLECTION_NAME, schema=schema, consistency_level="Strong")
    print("--> Collection еҲӣе»әжҲҗеҠҹгҖӮ")

    # еҲӣе»әзҙўеј•
    print("--> жӯЈеңЁдёәж–°йӣҶеҗҲеҲӣе»әзҙўеј•...")
    sparse_index = {"index_type": "SPARSE_INVERTED_INDEX", "metric_type": "IP"}
    collection.create_index("sparse_vector", sparse_index)
    print("зЁҖз–Ҹеҗ‘йҮҸзҙўеј•еҲӣе»әжҲҗеҠҹгҖӮ")

    dense_index = {"index_type": "AUTOINDEX", "metric_type": "IP"}
    collection.create_index("dense_vector", dense_index)
    print("еҜҶйӣҶеҗ‘йҮҸзҙўеј•еҲӣе»әжҲҗеҠҹгҖӮ")

collection = Collection(COLLECTION_NAME)
collection.load()
print(f"--> Collection '{COLLECTION_NAME}' е·ІеҠ иҪҪеҲ°еҶ…еӯҳгҖӮ")
```

**fields field type analysis:**

- **pk**: Primary key design, `auto_id=True` allows Milvus to automatically generate unique identifiers to avoid primary key conflicts
- **scalar fields**: 7 VARCHAR fields are used to store metadata, `max_length` optimizes storage according to actual data distribution
- **Sparse vector**: `SPARSE_FLOAT_VECTOR` type, stores keyword weights
- **Dense vector**: `FLOAT_VECTOR` type, fixed 1024 dimensions, stores semantic features


### 3.2 Step 2: BGE-M3 dual vector generation

BGE-M3 is used here as the vector generator, which is capable of generating both sparse vectors and dense vectors.

#### 3.2.1 Data loading and preprocessing

```python
if collection.is_empty:
    print(f"--> Collection дёәз©әпјҢејҖе§ӢжҸ’е…Ҙж•°жҚ®...")
```

#### 3.3.2 Hybrid retrieval execution

Use the RRF algorithm for hybrid retrieval, implemented through RRFRanker encapsulated in milvus. The core parameter of RRFRanker is the `k` value (default 60), which is used to control the smoothness of sorting in the RRF algorithm.

Among them, the larger the value of `k`, the smoother the ranking results; the smaller the value, the more prominent the weight of high ranking results.

```python
# е®ҡд№үжҗңзҙўеҸӮж•°
search_params = {"metric_type": "IP", "params": {}}

# е…Ҳжү§иЎҢеҚ•зӢ¬зҡ„жҗңзҙў
print("\n--- [еҚ•зӢ¬] еҜҶйӣҶеҗ‘йҮҸжҗңзҙўз»“жһң ---")
dense_results = collection.search(
    [dense_vec],
    anns_field="dense_vector",
    param=search_params,
    limit=top_k,
    expr=search_filter,
    output_fields=["title", "path", "description", "category", "location", "environment"]
)[0]

for i, hit in enumerate(dense_results):
    print(f"{i+1}. {hit.entity.get('title')} (Score: {hit.distance:.4f})")
    print(f"    и·Ҝеҫ„: {hit.entity.get('path')}")
    print(f"    жҸҸиҝ°: {hit.entity.get('description')[:100]}...")

print("\n--- [еҚ•зӢ¬] зЁҖз–Ҹеҗ‘йҮҸжҗңзҙўз»“жһң ---")
sparse_results = collection.search(
    [sparse_vec],
    anns_field="sparse_vector",
    param=search_params,
    limit=top_k,
    expr=search_filter,
    output_fields=["title", "path", "description", "category", "location", "environment"]
)[0]

for i, hit in enumerate(sparse_results):
    print(f"{i+1}. {hit.entity.get('title')} (Score: {hit.distance:.4f})")
    print(f"    и·Ҝеҫ„: {hit.entity.get('path')}")
    print(f"    жҸҸиҝ°: {hit.entity.get('description')[:100]}...")

print("\n--- [ж··еҗҲ] зЁҖз–Ҹ+еҜҶйӣҶеҗ‘йҮҸжҗңзҙўз»“жһң ---")
# еҲӣе»ә RRF иһҚеҗҲеҷЁ
rerank = RRFRanker(k=60)

# еҲӣе»әжҗңзҙўиҜ·жұӮ
dense_req = AnnSearchRequest([dense_vec], "dense_vector", search_params, limit=top_k)
sparse_req = AnnSearchRequest([sparse_vec], "sparse_vector", search_params, limit=top_k)

# жү§иЎҢж··еҗҲжҗңзҙў
results = collection.hybrid_search(
    [sparse_req, dense_req],
    rerank=rerank,
    limit=top_k,
    output_fields=["title", "path", "description", "category", "location", "environment"]
)[0]

# жү“еҚ°жңҖз»Ҳз»“жһң
for i, hit in enumerate(results):
    print(f"{i+1}. {hit.entity.get('title')} (Score: {hit.distance:.4f})")
    print(f"    и·Ҝеҫ„: {hit.entity.get('path')}")
    print(f"    жҸҸиҝ°: {hit.entity.get('description')[:100]}...")
```
